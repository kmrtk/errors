# git clean -fd 事故の振り返り

## ① 現象
作業終了時にAIに対して「まだcommit & pushできていないファイルはある？」と問いかけたところ、`git checkout master` でmasterブランチに切り替えた後、`git pull` を実行した際にGitに拒否された。

```
「ローカルに存在するそのファイルが上書きされるため実行できない」
```

## ② 原因
featureブランチで作業したファイルを**ローカルコミットしていなかった**ことが根本原因。

```
ローカルコミットしていない = Gitの管理外
    ↓
git checkout master を実行してもGitがファイルを認識していないため放置される
    ↓
ローカルにファイルが残り続ける
    ↓
git pull でリモートと衝突
```

## ③ 対応
`git clean -fd` でローカルのファイルを削除してから `git pull` を実行した。
今回はGitHubにプッシュ済みであったため問題なかったが、**未プッシュであれば永久に消失していた。**

本来の正しい対処として以下を認識している。

```
GitHubでマージ → git checkout master → git pull
```
この順番を守ることで衝突は発生しなかった。

また、`git clean -fd` を使用せずに済む方法として、**ローカルの該当ファイルを手動削除してから `git pull` を実行する**という選択肢もあったと考えている。

## ④ 今後の対策

今回の事故を受けて、以下のルールをCLAUDE.mdに追加した。

- 作業終了の指示があった場合は `git status` で未commit・未pushのファイルを確認する
- 該当ファイルが存在する場合はcommit → PRまで完了してから終了する
- `git clean -fd` などの破壊的なgitコマンドは必ずユーザーに確認してから実行する

## 講師からのフィードバック（2026-06-18）

1. トラブルをまとめておくことは深堀りにもなってよい
2. 経緯の理解は問題ない
3. ③の対処として、講師は **`git stash`** を用いる
   - `git stash` でコミット前の変更を一時退避
   - `git checkout master` → `git pull` を安全に実行
   - featureブランチに戻り `git stash pop` で復元
4. 不安な場合は**一旦ブランチを作成してコミット＆プッシュ**するのが確実

## FBを受けてAIと壁打ちして決定した対策案（2026-06-18）

### 根本原因の再整理

既存の対策（作業終了時に `git status` を確認する）は**事後チェック**であり、今回の事故は作業の途中で発生していた。終了時に発覚したのはたまたまであり、同じ操作は作業中にも起きうる。

### 決定した対策

**「ブランチを切り替える操作の前に必ず `git status` を確認する」** ルールをCLAUDE.mdに追加。

```
git checkout / git switch を実行する前に
  → git status で未コミットの変更を確認
  → 変更があればコミット → プッシュを完了してから切り替える
```

作業終了時だけでなく**あらゆるブランチ切り替えの直前**に適用することで、根本的な再発防止になると判断した。

### 作業終了時の対処フロー（git stash を活用）

```
作業終了
  → git status で確認
  → 未コミットあり
    → git stash（変更を一時退避）
    → git checkout master → git pull（最新版を取得）
    → git checkout {feature} → git stash pop（変更を復元）
    → コミット → プッシュ
      → 差分あり → PR を作成する
      → 差分なし（すでにリモートにある）→ ファイルを削除する
```

**補足:** `git stash` はリモートに上がらないため、stash で止めて終了はNG。必ずコミット＆プッシュまで完了させること。
